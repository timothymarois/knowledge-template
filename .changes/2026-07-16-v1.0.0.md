# Migration → 1.0.0 (2026-07-16)

**Breaking.** First versioned release. Migrates a project off the legacy (unversioned) doc scaffold — the
one with `.ai/docs/`, uppercase `PRD/` + `PRD-drafts/`, per-home `TEMPLATE.md` files, and `Owns` /
`Does NOT own` / `Core contract` / `Tests`-table PRDs.

If the repo still has a `.ai/docs/` folder and no `.knowledge/.version`, it is pre-1.0.0 and this whole
migration applies. Apply the steps in order, then set `.knowledge/.version` to `1.0.0` and run
`python3 .knowledge/scripts/doc-lint .knowledge`.

## What changed

- **`.knowledge/`, flat.** The agent-facing docs move from `.ai/docs/` to a flat `.knowledge/`; every home
  sits directly under it.
- **Catalog vs. rules split.** Each home's `README.md` is now a **catalog** (what's inside). The **rules**
  for writing live in `guides/docs-*.md`, with the template built in. All per-home `TEMPLATE.md` files are
  deleted.
- **`prd/` + isolated `prd-drafts/`.** Lowercase. A `prd/` contract may never cite a `prd-drafts/` id.
- **`references/`** is its own home (promoted out of `research/`).
- **PRD shape.** A single glyph table (`✅/❌ | ID | Requirement | Evidence`) replaces prose
  `Core contract` + a separate `Tests` table. Freshness is one `last_verified:` frontmatter date.
- **Catalogs are maintained by hand.** Nothing regenerates them; `doc-lint` fails the build when one falls
  behind its home.
- **The root `AGENTS.md` is checked** — it must exist and name the three orientation docs, since that load
  order is the only reason any of this gets read. Nothing else about the file is inspected.
- **Requirement IDs are an unbroken run.** A namespace numbers from one with no gaps. A contract holds what
  is true now, so a withdrawn requirement is erased rather than kept as struck history, and the rows below
  it renumber to close the gap — which means erasing a row is one change that also updates every citation
  of every ID at or after it, in the other contracts and in the code comments that name them.
- **Every requirement row stands alone**, and carries no infrastructure or vendor vocabulary — a row is read
  where it is cited, and a contract that names a database or a package is a schema description.
- **`OVERVIEW.md` says once that it describes the design**, and names the surfaces each audience touches.
- **`doc-lint`** enforces all of it in CI.

## Migration (apply in order)

1. **Move to `.knowledge/`.** Create `.knowledge/` and move everything from `.ai/docs/` into it (flattening
   the `docs/` layer away); delete the old `.ai/`. Update every `.ai/docs/...` path in `AGENTS.md`,
   `CODEMAP.md`, and code doc-blocks to `.knowledge/...`.
2. **Rename homes.** `PRD/` → `prd/`, `PRD-drafts/` → `prd-drafts/`. Promote `research/references/` to a
   top-level `references/`.
3. **Delete every `TEMPLATE.md`** under the homes. The template now lives inside each `guides/docs-*.md`.
4. **Add the guides.** Copy all `guides/docs-*.md` standards (prd, research, brief, codemap, memory) from
   this knowledge-template version into the project's `.knowledge/guides/`.
5. **Rewrite each home `README.md` as a catalog** (see this version's `template/.knowledge/*/README.md`):
   `prd/` = Components + Contents; `research/` = notes table; etc. **List everything actually in the home** —
   catalogs are maintained by hand, nothing regenerates them, and `doc-lint` fails on any file it can't find
   a row for (`prd/`, `prd-drafts/`, `research/`, `guides/`, and each `references/` set).
6. **Check the root `AGENTS.md`.** The lint requires one, naming all three of `.knowledge/BRIEF.md`,
   `.knowledge/CODEMAP.md`, and `.knowledge/MEMORY.md` — that load order is why any of these docs get read.
   Nothing else about the file is inspected. No `AGENTS.md`? Write one following
   `.knowledge/guides/docs-agents.md`.
7. **Convert each PRD** from the old shape to the glyph table:
   - Drop `Owns` / `Does NOT own` (they're the filename + citations) and the `Core contract` prose.
   - Turn each `R-<AREA>-n` bullet + its `Tests`-table row into one table row:
     `✅ | R-… | <requirement ≤25 words, no symbols> | <test name>`. Built-but-untested → `❌ … | file:line — no test`. Nothing yet → `❌ … | —`.
   - **Open the test the old table names before you write ✅.** If it no longer exists, or you cannot find
     it, the row is `❌`. The old `Tests` table was never checked by anything, so treat it as a claim to
     verify, not a fact to copy — carrying it across unread is how a migration manufactures a green
     contract for tests that were deleted years ago.
   - Add frontmatter `id` + `name`; add `last_verified` only once a real conformance review sets a ✅.
8. **Split multi-namespace files.** Any PRD carrying more than one `R-<NS>` becomes one file per namespace.
   Where a file restated another's requirement, **replace the restatement with a citation** to the single
   owning ID (this removes duplicate ID definitions).
9. **Rename filenames to `layer-node.md`** per the project's Components list, and reserve/keep IDs.
10. **Close every gap in a namespace's numbering.** The lint fails on a hole. **Before renumbering, grep the
    whole repo for the namespace** — contracts and code comments both cite these IDs, and a renumber
    silently re-points every citation at or after the gap. Where nothing cites them (a file mis-numbered
    from the start), renumber the rows down to close it. Where something does, update the citations in the
    same change. If a citation lives somewhere you cannot edit, leave the gap and ask the owner.
11. **Read every requirement row cold.** Restore the subject or verb that was trimmed to fit the word cap,
    and replace infrastructure and vendor nouns with the product behavior they protect. Recover the meaning
    from the test named in Evidence — never invent it. A row that won't fit in the cap once it stands alone
    is two requirements: raise it, don't split it.
12. **Add the two `OVERVIEW.md` elements:** the framing line under the title, and a `## What you use`
    section naming the surfaces. See `.knowledge/guides/docs-overview.md`.
13. **Stamp & lint.** Set `.knowledge/.version` to `1.0.0`; run `python3 .knowledge/scripts/doc-lint .knowledge`
   and fix every red (and `scripts/test_doc_lint.py` to confirm the linter itself is sound).

## Verify

- `doc-lint` exits clean, and `test_doc_lint.py` passes every case (the count grows as rules are added — take the shipped number, not one written here).
- No `prd/` file cites a `prd-drafts/` id; every namespace lives in exactly one file; no ID defined twice.
- Each home `README.md` is a well-formed catalog listing everything actually in that home.
- The root `AGENTS.md` exists and names all three orientation docs.
