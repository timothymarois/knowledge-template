# Migration → 2.0.0 (2026-07-19)

**Breaking.** Three new or widened checks fail docs that passed under 1.0.x, and one guide's placement rule
is reworded. Apply the steps in order, then set `.knowledge/.version` to `2.0.0` and run
`python3 .knowledge/scripts/doc-lint .knowledge`.

## What changed

- **Catalogs are honest now.** Every home catalog said *"Contents — generated / Regenerated by `doc-lint`"*.
  Nothing was ever regenerated — and only `prd/` was even checked for completeness. Agents read
  "regenerated" and stopped maintaining the catalogs by hand, so they drifted silently. The wording now says
  **maintained by hand**, and the completeness check covers **every** home: `prd/`, `prd-drafts/`,
  `research/`, `guides/` (each non-README `.md`) and `references/` (each set subfolder).
- **`AGENTS.md` is checked.** The `.knowledge/` load order in the repo's root `AGENTS.md` is the only reason
  an agent reads any of these docs, and nothing guarded it. Two shallow checks: the file exists, and it names
  all three of `.knowledge/BRIEF.md`, `.knowledge/CODEMAP.md`, `.knowledge/MEMORY.md`. **Nothing else about
  the file is inspected** — not sections, not wording, not order.
- **`--payload` flag.** `doc-lint --payload <path>` skips the `AGENTS.md` check, for linting a payload that
  isn't inside an adopting repo. Adopting projects never need it.
- **One file per namespace is enforced.** The standard always said it; the linter only caught duplicate
  *IDs*. Two files both declaring `id: THING` with disjoint numbers now fail.
- **The PRD placement ladder is stack-neutral.** It asked whether a requirement "names a placeable thing" —
  spatial vocabulary that means nothing in a web app, a service, or an extension. It now reads against
  whatever components the project declared. Wording only; no requirement moves.

## Migration (apply in order)

1. **Replace the shipped files.** Copy this version's `template/.knowledge/scripts/doc-lint`,
   `scripts/test_doc_lint.py`, and all of `guides/docs-*.md` over the project's copies.
2. **Reword the catalog headings.** In each of `prd/README.md`, `prd-drafts/README.md`, `research/README.md`,
   `references/README.md`, `guides/README.md`, replace the "— generated / Regenerated by `doc-lint`" line
   with "— maintained by hand / Add a row when you add a `<thing>`; `doc-lint` fails the build if one is
   missing." (Keep the `## Contents` / `## Notes` / `## Drafts` / `## Sets` heading prefix — it's checked.)
3. **Fill the catalogs in.** Run `doc-lint`; for every `catalog omits <name>`, add the row. This is where the
   drift that accumulated under 1.0.x surfaces — expect notes, guides, and reference sets that were never
   listed.
4. **Check `AGENTS.md`.** Make sure the repo root has one and that its *Before you work* section still names
   `.knowledge/BRIEF.md`, `.knowledge/CODEMAP.md`, and `.knowledge/MEMORY.md`. If the repo has no
   `AGENTS.md`, write one following `.knowledge/guides/docs-agents.md`.
5. **Split any shared namespace.** For every `namespace ... is already owned by`, decide which file owns the
   namespace and give the other its own — the IDs it already defined move with it, unchanged.
6. **Stamp & lint.** Set `.knowledge/.version` to `2.0.0`; run `python3 .knowledge/scripts/doc-lint
   .knowledge` and `python3 .knowledge/scripts/test_doc_lint.py`, and fix every red.
