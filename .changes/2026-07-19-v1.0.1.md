# Migration → 1.0.1 (2026-07-19)

**Patch — fixes a shipped bug that turns a project's own CI red.** No doc changes; one script is replaced.

## What changed

`scripts/test_doc_lint.py` builds its "valid base" by copying the live `.knowledge/` it ships inside, then
overwriting `prd/README.md` with a two-component fixture catalog. In 1.0.0 it did **not** clear the
project's real content first — so as soon as a project wrote its first PRD, that PRD survived the copy,
the fixture catalog didn't list it, and the valid base failed the lint. Result: `test_doc_lint.py` exits
non-zero (`catalog Contents omits <file>`, `filename prefix not a listed component`) in any project that
has actually used the system, even though the project's own docs are perfectly valid.

`build_base()` now clears `prd/`, `prd-drafts/`, `research/`, and `references/` of everything but their
`README.md` after copying, so the fixture is the whole of the valid base. The shipped *structure* is still
copied from the real payload, so the structural checks keep testing the real tree.

## Migration (apply in order)

1. **Replace one file.** Copy this version's `template/.knowledge/scripts/test_doc_lint.py` over the
   project's `.knowledge/scripts/test_doc_lint.py`. Nothing else changes; no doc edits are required.
2. **Stamp.** Set `.knowledge/.version` to `1.0.1`.

## Verify

- `python3 .knowledge/scripts/test_doc_lint.py` prints `23/23 checks passed` — including in a project that
  already has PRDs in `prd/` (before this fix, that project saw `22/23` with the valid-base case failing).
- `python3 .knowledge/scripts/doc-lint .knowledge` still exits clean.
